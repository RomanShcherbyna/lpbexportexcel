# Product Import & AI Extensions – Design

> **Назначение файла**  
> Это главный документ по расширениям импорта (SKU, ИИ, валюты, категории и т.д.).  
> Все новые фичи вокруг импорта/ИИ нужно сверять с этим файлом и по возможности держать в синхроне.

> **Язык интерфейса**  
> Весь UI для этого пайплайна (страницы импорта, маппинга, генерации, проверки и итоговой таблицы) должен быть **на русском языке**, с понятными текстами кнопок/подсказок для не‑технического пользователя.

---

## 0. Общая картина

- Источник: Excel/CSV от поставщика.
- Шаги:
  1. Загрузка и выбор поставщика.
  2. Маппинг колонок поставщика → наш шаблон.
  3. Нормализация / ИИ‑предложения (категории, переводы и т.п.).
  4. Валидатор.
  5. Просмотр результата, ручное редактирование.
  6. Экспорт CSV (и/или XLSX).
- Важное правило: **если данные уже есть во входном файле, мы их не придумываем заново** (для SKU, имен, категорий и т.д.).

Все описанные ниже штуки (ИИ, формулы, SKU, валюты) должны:

- иметь управление через UI сайта;
- работать пакетно (батчами), чтобы можно было:
  - выбрать, какие поля генерировать;
  - отправить одну или несколько задач;
  - получить результат, посмотреть и только потом применить.

---

## 1. Product name (PL)

### 1.1. Бизнес‑правило

- `Product name (PL)` — **прямой перевод** `Product name (EN)` (без выдуманных свойств).
- Если `Product name (PL)` уже есть во входе или пользователь вписал руками → **не перетирать**.

### 1.2. Как генерируется

- Генерация через ИИ по запросу:
  - вход: `Product name (EN)` + контекст (`Brand`, `Gender`, категории и т.д.);
  - выход: одна строка, которую можно отредактировать перед применением.

### 1.3. Управление через UI

- На сайте должна быть панель (например, “Post‑processing / AI helpers”), где можно:
  - увидеть список полей, доступных для генерации (галочки):
    - `Product name (PL)`
    - категории
    - цвета
    - и т.п.;
  - выбрать, **что именно генератор будет трогать** в этом запуске.
- Для `Product name (PL)`:
  - опция “Сгенерировать PL‑названия для всех строк, где PL пустой”.
  - батч‑запуск (один запрос на job / группу строк, а не по одной — экономия).
  - после возврата результата:
    - показать различия (EN / сгенерированный PL);
    - дать возможность:
      - принять всё сразу;
      - принять только выбранные строки;
      - отредактировать отдельные значения.

---

## 2. SKU – формула + таблица соответствий

### 2.1. Правила

1. Если во входном файле колонка `Sku` замаплена и строка содержит значение:
   - использовать это значение как есть;
   - **ни формулу, ни генерацию, ни ИИ к этим строкам не применять**.

2. Если `Sku` пустой:
   - генерируем детерминированно **без ИИ**;
   - опорные поля для ключа (минимум 2–3):
     - `Brand` (`Brend  `)
     - `Supplier Product ID` / `Style No`
     - `Color Code`
     - размер (`Shoe size` / `Age Size` / `Height Size` — по поставщику);
   - ключ должен быть **устойчивым** — при повторном импорте тех же данных даёт тот же результат.

### 2.2. Формула (пример)

- Нормализация входных значений (для ключа и SKU):
  - `mb_strtoupper`,
  - обрезать пробелы по краям,
  - внутри пробелы → `-`,
  - разрешить только `[A-Z0-9-]`.

- Пример составного ключа:

```text
KEY = BRAND + '|' + SUPPLIER_PRODUCT_ID + '|' + COLOR_CODE + '|' + SIZE
```

- Пример SKU по умолчанию:

```text
SKU = BRANDSHORT + '-' + SUPPLIER_PRODUCT_ID + '-' + COLOR_CODE + '-' + SIZE
```

где `BRANDSHORT` — первые 3–4 нормализованные буквы бренда.

### 2.3. Таблица соответствий (кеш)

- Отдельная таблица в БД (идея): `sku_identity_map`
  - `id`
  - `composite_key` (строка, например `BRAND|SUPPLIER_PRODUCT_ID|COLOR|SIZE`)
  - `sku`
  - timestamps.
  - индекс по `composite_key`.

- Алгоритм для строки без `Sku`:
  1. Собрать `composite_key`.
  2. Поиск в `sku_identity_map`:
     - если найден → взять `sku`;
     - если нет → сгенерировать новым правилом, сохранить и использовать.

- Так система:
  - не трогает уже заданные вручную `Sku`;
  - даёт стабильные коды при повторном импорте;
  - не создаёт дубликаты для тех же сочетаний атрибутов.

---

## 3. Категории и саб‑категории через ИИ

### 3.1. Данные

Для ИИ‑классификации использоватся:

- `Product name (EN)` / `(PL)`
- `Brand`
- `Type` / `item_category` / `item SubCategory` от поставщика
- `Gender`
- размер (`Age/Shoe/Height Size`)
- опционально: `Description EN`.

### 3.2. Справочник категорий

- В БД отдельные таблицы/конфиги:
  - `our_category` (основная категория);
  - `our_subcategory` (сабкатегория).
- Использовать стабильные ID/slug для хранения и человекочитаемые названия для UI.

### 3.3. Логика ИИ

- Если SKU уже есть в таблице `sku_category_map` (или аналоге) → использовать сохранённые значения, ИИ не дергать.
- Для новых SKU:
  - ИИ возвращает:
    - `our_category`
    - `our_subcategory`
    - `confidence` (0–1)
    - короткий `reason`.

- Порог уверенности:
  - если `confidence ≥ threshold` (например 0.85) → можно авто‑принять;
  - иначе → пометить как требующие проверки.

### 3.4. UI и батчи

- В UI должен быть инструмент:
  - выбрать **какие именно поля генерировать** (в данном случае — категории/сабкатегории);
  - выбрать набор строк (по job, по SKU, по фильтру без категории);
  - запустить одну батч‑задачу ИИ.

- После выполнения:
  - показать таблицу с предложениями;
  - дать пользователю:
    - принять всё;
    - принять по строкам;
    - при необходимости отредактировать;
  - записать утверждённые значения в постоянную таблицу (например, `sku_category_map`).

---

## 4. Поле «Написано на этикетке товара»

### 4.1. Формульный подход

- Для каждого бренда в конфиге задаётся шаблон (формула), например (на примере Liewood):

```text
"{STYLE_NAME} {COLOR_DESCRIPTION} {SIZE} BATCH NO: {BATCH_NO}"
```

- При экспорте:
  - подставлять реальные значения;
  - если каких‑то полей нет → либо опускать сегмент, либо оставлять пусто (настраивается).

### 4.2. Отложенная ИИ‑версия

- Можно позже добавить ИИ‑генерацию “читаемых” этикеток, но **по тем же полям**, без выдуманных характеристик.

---

## 5. Цвета (EN → PL и из кодов)

### 5.1. Словарь

- Таблица `color_map`:
  - `brand` (опц.)
  - `source_value` (как у поставщика: “Tuscany rose multi mix” или `2696`)
  - `target_pl` (строка для вывода)
  - `normalized_source` (для поиска).

- При импорте:
  - ищем по `source_value` и/или `Color Code`;
  - если находим → заполняем `Color Description PL`.

### 5.2. ИИ для новых значений

- Если в словаре нет:
  - вызываем ИИ (по необходимости и по галочке в UI) для перевода:
    - вход: исходный цвет EN +, при необходимости, контекст;
    - выход: строка для `Color Description PL`.
  - новое соответствие может:
    - либо сразу добавляться в `color_map`,
    - либо сначала появляться в статусе “draft” и требовать ручного подтверждения.

---

## 6. Курсы валют и пересчёт цен (без ИИ)

### 6.1. Настройки

- В админке нужны настройки:
  - текущие курсы (например, `EUR→PLN`, `USD→PLN`);
  - предпочтения округления:
    - шаг (0.01, 0.10, 1.00 и т.д.);
    - направление (вверх/вниз/до ближайшего).

### 6.2. Формулы

- Для каждой строки:
  - если есть `Wholesale Price EUR`:

    ```text
    Wholesale Price PLN = ROUND(Wholesale Price EUR * курс_EUR_PLN, по настройкам)
    ```

  - если есть `Recommended Retail Price EUR`:

    ```text
    Recommended Retail Price PLN = ROUND(RRP_EUR * курс_EUR_PLN, по настройкам)
    ```

- Желательно сохранять курс, по которому посчитано (например, в служебном JSON‑поле).

---

## 7. Общие требования к UI и батч‑генерации

1. **Выбор полей для генерации**
   - На уровне job или всего импорта должна быть форма:
     - чекбоксы по полям:
       - `Product name (PL)`
       - категории/сабкатегории
       - цвета
       - “этикетка”
       - др.
     - таким образом, пользователь может решить:
       - что именно генерить в конкретном запуске;
       - не тратить лимит ИИ на поля, которые сейчас не нужны.

2. **Батчи (одна или несколько задач)**
   - Запуск ИИ/формульных генераций всегда делается:
     - по выбранным полям,
     - по выбранному набору строк.
   - Внутри бекенда это может разбиваться на несколько запросов к ИИ, но с точки зрения пользователя это одна операция.

3. **Предложение акцепта**
   - После возвращения результата:
     - показывается черновой вариант данных:
       - старое значение (если было),
       - новое предложенное.
     - Нельзя сразу переписывать итоговую таблицу без возможности review.

4. **Редактор перед скачиванием**
   - На странице результата (перед “Download CSV”):
     - дать удобную таблицу:
       - фильтры (показывать только строки, где что‑то сгенерировано / есть проблемы);
       - массовое удаление/очистка значений в выбранных колонках;
       - ручное редактирование ячеек.
   - После сохранения изменений в превью:
     - экспорт CSV/XLSX должен использовать уже отредактированные данные.

5. **Прозрачность и контроль**
   - Пользователь всегда видит:
     - какие поля были сгенерированы ИИ/формулами;
     - какие были заданы исходно поставщиком;
     - может откатить/очистить выбранные колонки.

## 8. Приоритет внедрения

1. SKU: формула + таблица `sku_identity_map` + логика “если Sku есть во входе — не трогаем”.
2. Курсы валют и пересчёт цен (UI + формулы).
3. Категории/сабкатегории через ИИ + таблица соответствий + UI.
4. Цвета (словари + ИИ).
5. Product name (PL) через ИИ.
6. Формулы для “Написано на этикетке товара”.

При изменении бизнес‑логики импорта обязательно обновлять этот файл и дублировать ключевые моменты в пользовательской документации (LPB/README.md или UI‑подсказках).

## 9. Последующий этап: реализация русского UI

После того как все пункты этого дизайна (1–8) будут реализованы и протестированы на уровне бэкенда и пайплайна данных, нужно выполнить **отдельную UI‑задачу**:

- Сделать полный интерфейс импорта/обработки **на русском языке**:
  - главная страница импорта (загрузка файла, выбор поставщика);
  - шаг маппинга (наши поля сверху вниз, столбец “Final selected”, снизу — предпросмотр первых строк файла поставщика);
  - шаг выбора генераций (чекбоксы “что генерить / не генерить”);
  - экран статуса фоновой обработки (job на сервере, чтобы прогресс не терялся при перезагрузке);
  - экран проверки и акцепта ИИ/формул (таблицы “до/после” с галочками);
  - итоговый редактор таблицы перед скачиванием (фильтры, инлайн‑редактирование, массовые очистки полей).

- Все подписи/кнопки/подсказки должны быть:
  - **формулированы по‑русски** и понятны человеку, который не разработчик;
  - согласованы с этим файлом (чтобы легко находить, какой экран за что отвечает).

Эта UI‑задача должна планироваться **после** завершения и стабилизации логики, описанной выше, и ссылаться на `LPB/PRODUCT_PIPELINE_DESIGN.md` как на основной источник требований.

## Product Import Pipeline – расширения (SKU, ИИ, категории, валюты)

### Цели

- Минимум ручной работы при импорте прайсов от поставщиков.
- Жёсткая предсказуемость там, где это критично (SKU, цены, ID).
- ИИ помогает, но **никогда не ломает** данные молча: всегда есть просмотр/подтверждение.
- Все настройки и инструменты доступны через UI сайта.

---

### 1. Product name (PL)

**Правило**

- `Product name (PL)` = **перевод 1:1** из `Product name (EN)`.
- Никаких фантазий, добавления свойств “от себя” и т.п.

**Когда вызывать**

- После мэппинга в шаблон.
- Если `Product name (PL)` уже заполнен → не трогаем.
- Если пустой → отправляем в ИИ, вход: `Product name (EN)` + контекст (Brand, категория, цвет и т.п.), выход: одна строка для PL.

**UI**

- В превью результата показывать EN/PL рядом.
- Возможность фильтровать строки без PL, редактировать и перегенерировать.

---

### 2. SKU – формула + таблица соответствий

**Базовая идея**

- SKU стабильно идентифицирует вариант товара и воспроизводится при повторном импорте.

**Правила**

1. Если во входе есть колонка `Sku` и она замаплена:
   - использовать как есть,
   - **не** применять формулу/генерацию.

2. Если `Sku` пустой:
   - генерировать по формуле из стабильных полей (Brand, Supplier Product ID/Style No, Color Code, Size),
   - использовать как минимум 2–3 поля в ключе (минимум три: бренд + ID + цвет/размер).

**Формула (пример)**

Нормализация:

- всё в верхний регистр;
- пробелы → `-`;
- убрать лишние спецсимволы.

Ключ:

```text
KEY = BRAND + '|' + SUPPLIER_PRODUCT_ID + '|' + COLOR_CODE + '|' + SIZE
```

SKU по умолчанию (если нет сохранённого):

```text
SKU = BRANDSHORT + '-' + SUPPLIER_PRODUCT_ID + '-' + COLOR_CODE + '-' + SIZE
```

где `BRANDSHORT` — первые 3–4 буквы бренда.

**Таблица соответствий (`sku_identity_map`)**

- Поля:
  - `composite_key` (строка — нормализованный KEY),
  - `sku`,
  - `created_at`, `updated_at`.
- Индекс по `composite_key`.

Алгоритм:

1. Собираем KEY.
2. Ищем KEY в `sku_identity_map`:
   - найден → берём существующий SKU;
   - нет → генерируем SKU формулой, сохраняем в таблицу, используем.

---

### 3. Категория и сабкатегория через ИИ

**Справочник**

- Отдельная таблица категорий/подкатегорий с человекопонятными ID/slug.

**Вход в ИИ**

- Product name (EN/PL), Brand, поставщикские category/type, Gender, размер, части Description EN.

**Задача ИИ**

- Вернуть: `our_category`, `our_subcategory`, `confidence`, `reason`.

**Применение**

- Если SKU уже есть в `sku_category_map` → использовать существующее соответствие, ИИ не звать.
- Для новых:
  - `confidence >= threshold` (напр. 0.85) → можно применять автоматически,
  - ниже порога → требуется ручное подтверждение в UI.

**UI**

- Таблица: SKU, имя, поставщикская категория, предложенная категория/сабкатегория, confidence, reason.
- Фильтры по “сомнительным” строкам.
- Возможность массово принять/переопределить.

Все подтверждённые значения хранить в `sku_category_map` (sku, our_category, our_subcategory, source, confirmed_at).

---

### 4. “Написано на этикетке товара” (формулы)

**Идея**

- Для каждого бренда задаётся строковый шаблон, который собирает поля в нужный текст.

Пример для Liewood:

```text
LABEL = STYLE_NAME + ' ' + COLOR_DESCRIPTION + ' ' + SIZE + ' BATCH NO: ' + BATCH_NO
```

Где поля берутся из:

- `Product name (EN)` / `Style Name`,
- `Color Description EN`,
- `Size` (Shoe size / Age Size / Height Size),
- `Batch No` (если есть).

Хранить шаблоны лучше в конфиге/БД, чтобы можно было редактировать без правок кода.

---

### 5. Цвета (EN → PL, коды → названия)

**Словарь + ИИ**

1. Таблица `color_map`:
   - `brand` (опционально),
   - `source_value` (строка от поставщика: описание или код),
   - `target_pl` (наш перевод),
   - `normalized_source`.
2. Алгоритм:
   - Сначала ищем `source_value` в `color_map`.
   - Если нет:
     - вызываем ИИ с промптом “Переведи цвет одежды на польский без лишних слов”,
     - результат пишем в `Color Description PL` и, при необходимости, в `color_map` после ревью.

При коде цвета (цифры):

- ищем по коду,
- если нет — используем описание на EN, код просто копируем.

---

### 6. Курсы валют и пересчёт цен

**Настройки**

- В админке — страница “Финансы / Курсы”:
  - текущие курсы (EUR→PLN и т.п.),
  - дата/время курса,
  - тип округления (до 0.01/0.10/1.00, вверх/вниз/до ближайшего).

**Формулы**

- `Wholesale Price PLN = ROUND(Wholesale Price EUR * курс, правило_округления)`
- `RRP PLN = ROUND(RRP EUR * курс, правило_округления)`

Хранить использованный курс в метаданных (`exchange_rate_used`).

---

### 7. UI: общая панель управления

Нужно несколько страниц в админке:

1. **Список импортов/джобов** – статус, поставщик, кнопки открытия превью, перегенерации PL, ИИ‑категорий, пересчёта цен.
2. **SKU / идентичность** – просмотр и редактирование `sku_identity_map`.
3. **Категории** – справочник категорий + `sku_category_map` (исправление, перезапуск ИИ).
4. **Цвета** – `color_map` (редактирование, подтверждение ИИ‑переводов).
5. **Курсы валют** – управление курсами и округлением.
6. **Превью таблиц** – полноэкранный просмотр результата импорта с фильтрами и инлайн‑редактированием.

---

### 8. Приоритет внедрения

1. SKU: формула + `sku_identity_map` + правило “если Sku уже есть — не трогать”.
2. Пересчёт валют (формулы + UI настроек).
3. ИИ‑категории + `sku_category_map`.
4. Цвета EN→PL (словарь + ИИ).
5. Product name (PL) через ИИ.
6. Формулы для “Написано на этикетке товара” по брендам.

> При изменении логики **сначала обновлять здесь**, потом реализовывать в коде и UI.

# Product Import Pipeline – расширения (SKU, ИИ, категории, валюты)

## Цели

- Минимум ручной работы при импорте прайсов от поставщиков.
- Жёсткая предсказуемость там, где это критично (SKU, цены, ID).
- ИИ помогает, но **никогда не ломает** данные молча: всегда есть просмотр/подтверждение.
- Все настройки и инструменты доступны через UI сайта.

---

## 1. Product name (PL)

### 1.1. Правило

- `Product name (PL)` = **перевод 1:1** из `Product name (EN)`.
- Никаких фантазий, добавления свойств “от себя” и т.п.

### 1.2. Где и как вызывать

- Этап: после того как данные уже промапились в базовый шаблон колонок.
- Для каждой строки:
  - Если `Product name (PL)` **уже заполнен** → не трогаем.
  - Если пустой → отправляем в ИИ.

Вход ИИ:

- `Product name (EN)` (обязательно)
- дополнительный контекст: `Brand`, `Gender`, `item_category`, `item SubCategory`, цвет.

Выход ИИ:

- одна строка для `Product name (PL)`.

### 1.3. UI

- В финальном превью (страница результата):
  - Показать обе колонки: `Product name (EN)` и `Product name (PL)`.
  - Возможности:
    - фильтровать строки с пустым/сгенерированным PL;
    - редактировать вручную прямо в таблице;
    - перегенерировать PL для выбранных строк.

---

## 2. SKU – формула + таблица соответствий

### 2.1. Базовая идея

- SKU должен:
  - стабильно идентифицировать **вариант товара** (цвет/размер);
  - воспроизводиться при повторном импорте тех же данных.

### 2.2. Правила заполнения

1. **Если во входной таблице есть колонка `Sku` и она замаплена →**
   - используем её **как есть**;
   - **никаких формул/генерации для этих строк не делаем**.

2. **Если `Sku` пустой → используем генерацию:**
   - Опорные поля (примерный набор, можно уточнить для каждого поставщика):
     - `Brand` / `Brend  `
     - `Supplier Product ID` (Style No)
     - `Color Code`
     - размер (например, `Shoe size` или `Age Size` или `Height Size`)
   - Требование: использовать **минимум 2–3 поля как ключ** (чтобы различать варианты).

### 2.3. Формула (пример)

- Нормализация:
  - все буквы в верхний регистр;
  - пробелы → `-`;
  - убрать спецсимволы, кроме `-` и цифр.

- Пример ключа:

```text
KEY = BRAND + '|' + SUPPLIER_PRODUCT_ID + '|' + COLOR_CODE + '|' + SIZE
```

- SKU по умолчанию (если нет старого значения в таблице соответствий):

```text
SKU = BRANDSHORT + '-' + SUPPLIER_PRODUCT_ID + '-' + COLOR_CODE + '-' + SIZE
```

где `BRANDSHORT` = первые 3–4 буквы бренда, нормализованные.

### 2.4. Таблица соответствий (кеш)

Отдельная сущность в базе, например `sku_identity_map`:

- Поля:
  - `id`;
  - `composite_key` (строка, например JSON или `BRAND|SUPPLIER_PRODUCT_ID|COLOR|SIZE`);
  - `sku` (строка);
  - `created_at`, `updated_at`.
- Индекс по `composite_key`.

Алгоритм при импорте строки без `Sku`:

1. Собрать `composite_key`.
2. Искать его в `sku_identity_map`:
   - если найден → взять `sku` оттуда;
   - если нет:
     - сгенерировать `sku` формулой;
     - сохранить в `sku_identity_map`;
     - использовать его для этой строки.

Следствия:

- При повторном импорте той же таблицы или обновлённого прайса всегда получаем **тот же SKU**, если ключевые поля не изменились.
- Если поставщик чуть изменил данные (например, лишний пробел в имени), но ключ отнормирован — всё равно найдём старый SKU.

---

## 3. Категория и сабкатегория через ИИ

### 3.1. Справочник категорий

- В базе/конфиге хранить полный список:
  - `our_category` (например, “Одяг”, “Взуття”);
  - `our_subcategory` (например, “Футболки”, “Сандалии”, “Кепки”).
- Желательно иметь человекопонятный ID (строка/slug) + i18n имена.

### 3.2. Входные данные для ИИ

Для каждого артикула:

- `Product name (EN)` / `(PL)`;
- `Brand`;
- `Type` / `item_category` / `item SubCategory` от поставщика;
- `Gender`;
- информация о размере (Age/Shoe size);
- возможно: часть `Description EN`.

### 3.3. Задача ИИ

- Вернуть:
  - `our_category`;
  - `our_subcategory`;
  - `confidence` (0–1);
  - `reason` (краткое объяснение).

### 3.4. Правила применения

- Если этот SKU **уже виделся** ранее и для него сохранена категория:
  - не дергать ИИ, просто брать сохранённое соответствие.

- Для новых SKU:
  - Если `confidence` ≥ порога (например, 0.85):
    - можно **автоматически подставлять** категорию/сабкатегорию;
    - помечать строку как “auto”.
  - Если ниже порога:
    - показывать в UI как “требует проверки”;
    - давать пользователю выбрать manually.

### 3.5. UI

- Страница нормализации категорий:
  - Таблица: `Sku`, `Product name`, `Поставщик category`, `Предложенная category`, `Предложенная subcategory`, `confidence`, `reason`.
  - Возможности:
    - фильтр: только “сомнительные” (ниже порога);
    - массово принять предложенные для выбранных строк;
    - вручную выбрать категорию/сабкатегорию из выпадающего списка.

- Сохранение:
  - Все подтверждённые варианты сохраняются в таблицу `sku_category_map`:
    - `sku`;
    - `our_category`;
    - `our_subcategory`;
    - `source` (ai/manual);
    - `confirmed_at`.

---

## 4. Поле “Написано на этикетке товара” (формула)

### 4.1. Базовое правило (на примере Liewood)

Из строки `"Описание"` в шаблоне:

- Формула вида:

```text
LABEL = STYLE_NAME + ' ' + COLOR_DESCRIPTION + ' ' + SIZE + ' BATCH NO: ' + BATCH_NO
```

- Поля-источники:
  - `Product name (EN)` или `Style Name`;
  - `Color Description EN`;
  - `Size` (Shoe size / Age Size / Height Size – по бренду);
  - `Batch No` (если есть или отдельное поле).

### 4.2. Общий подход

- Для каждого бренда завести **формулу** в конфиге (PHP или БД), вместо жёстко пришитого кода.

Примеры:

- `"{STYLE_NAME} {COLOR} {SIZE}"`;
- `"BELLO STRIPE BIKINI {STYLE_NO} COLOR: {COLOR_CODE} SIZE {SIZE}"`.

- На этапе экспорта:
  - подставлять значения в шаблон;
  - если какого‑то поля нет — либо пропускать сегмент, либо оставлять пусто (правило настраиваемое).

---

## 5. Цвета (EN → PL, коды → названия)

### 5.1. Двоичный подход: словарь + ИИ

1. **Словарь:**
   - Таблица `color_map`:
     - `brand` (опц.);
     - `source_value` (то, что приходит от поставщика — например, “Tuscany rose multi mix” или `2696`);
     - `target_pl` (как хотим видеть по‑польски/русски);
     - `normalized_source` (для быстрого поиска).
   - При импорте:
     - если `Color Description EN` или `Color Code` есть в словаре → используем словарь.

2. **ИИ:**
   - Если в словаре нет:
     - вызвать ИИ с промптом вида: “Переведи цвет одежды на польский, без лишних слов. Вход: 'Peppermint multi mix'.”
     - результат записать:
       - в `Color Description PL` для конкретной строки;
       - и **опционально** добавить в `color_map` как новое соответствие (после ручного review).

### 5.2. Обработка кодов

- Если цвет задаётся числом (`Color Code`: `2696`):
  - сначала ищем `Color Code` в словаре;
  - если нет — ИИ работает на уровне `Color Description EN`, код просто копируем как есть.

---

## 6. Курсы валют и пересчёт цен

### 6.1. Настройки

- В UI сайта нужна страница “Финансы / Курсы”:
  - `курс_EUR_PLN`, `курс_USD_PLN` и т.д.;
  - дата/время курса;
  - стратегия округления:
    - до 0.01;
    - до 0.10;
    - до 1.00 (полный злотый);
  - направление:
    - вверх (`ceil`);
    - вниз (`floor`);
    - до ближайшего (`round`).

### 6.2. Формулы

- Для каждой строки:
  - Если есть `Wholesale Price EUR`:
    - `Wholesale Price PLN = ROUND(Wholesale Price EUR * курс_EUR_PLN, правила_округления)`.
  - Если есть `Recommended Retail Price EUR`:
    - `Recommended Retail Price PLN = ROUND(RRP_EUR * курс_EUR_PLN, правила_округления)`.

- Важно:
  - сохранять курс, по которому считали, в служебном поле (например, в JSON‑метаданных `exchange_rate_used`).

## 7. Требования к UI (общая панель управления)

Чтобы всем этим было **реально удобно пользоваться**, нужен блок страниц в админке.

### 7.1. Список импортов / джобов

- Статус, поставщик, дата, кол-во строк.
- Кнопки:
  - открыть превью;
  - запустить/перезапустить генерацию PL‑названий;
  - запустить ИИ‑категоризацию;
  - пересчитать цены по новому курсу (опционально).

### 7.2. Панель SKU / идентичности

- Таблица `sku_identity_map`:
  - ключ атрибутов;
  - SKU;
  - когда создан.
- Возможности:
  - найти по SKU/ключу;
  - вручную исправить SKU для конкретного ключа (на будущее);
  - увидеть, какие товары этим SKU уже пользуются.

### 7.3. Категории

- Справочник категорий и сабкатегорий.
- Таблица `sku_category_map`:
  - SKU;
  - категория/подкатегория;
  - источник (ai/manual);
  - дата подтверждения.
- Запуск ИИ‑сегрегации для выбранных SKU.

### 7.4. Цвета

- Таблица `color_map`:
  - `brand` (опц.), `source_value`, `target_pl`, статус (draft/confirmed).
- Возможности:
  - добавлять/редактировать руками;
  - видеть, какие значения были сгенерированы ИИ и требуют проверки.

### 7.5. Курсы валют

- Форма с текущими курсами и правилами округления.
- История изменений.

### 7.6. Просмотр итоговых таблиц

- Страница превью результата импорта:
  - фильтры по проблемам (нет PL, нет SKU, нет категории, нет цвета PL, нет цены в PLN);
  - инлайн‑редактирование полей;
  - скачивание CSV с уже сгенерированными полями.

2. Пересчёт валют (формулы + UI настроек).
3. ИИ‑категории + `sku_category_map`.
4. Цвета EN→PL (словарь + ИИ).
5. Product name (PL) через ИИ.
